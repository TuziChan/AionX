## 2. 关键风险与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| Rust 通道 SDK 不成熟 | Telegram/Lark/DingTalk 集成困难 | 中 | 使用 reqwest 直接调用 HTTP API，或通过 Sidecar 运行 Node.js 脚本 |
| ACP 协议变更 | Agent 通信中断 | 低 | 抽象层隔离，快速适配 |
| WebView 兼容性 | 不同平台渲染差异 | 中 | 使用标准 Web API，避免平台特有特性 |
| 扩展沙箱安全 | 恶意扩展风险 | 中 | Wasm 沙箱或受限 JS 运行时 |
| 大型对话性能 | 消息过多导致卡顿 | 中 | 虚拟滚动 + 分页加载 |

## 3. 技术决策记录

### TD-01: 状态管理选择 Zustand 而非 Jotai/Valtio
- Zustand API 最简洁，学习成本最低
- 与 React 19 兼容性好
- 内置 persist 中间件，适合配置持久化
- 社区活跃，生态成熟

### TD-02: WebUI 服务器选择 Axum 而非 Actix-web
- Axum 基于 tokio + tower，与 Tauri 的 tokio 运行时共享
- API 设计更现代，类型安全更好
- 中间件系统（tower）更灵活
- 性能与 Actix-web 相当

### TD-03: 密码哈希选择 Argon2 而非 bcrypt
- Argon2 是 Password Hashing Competition 获胜者
- 抗 GPU/ASIC 攻击能力更强
- Rust 原生实现，无 FFI 开销
- 向后兼容：可同时支持验证旧 bcrypt 哈希

### TD-04: Agent 通信选择子进程 stdin/stdout 而非 HTTP
- 与原 Electron 版本保持一致
- 无需额外端口占用
- 进程生命周期管理更简单
- JSON-RPC 协议清晰

### TD-05: 通道平台 Rust 实现策略
- Telegram: 使用 teloxide crate（成熟的 Rust Telegram Bot 框架）
- 飞书/钉钉: 使用 reqwest 直接调用 HTTP API（无成熟 Rust SDK）
- 如遇阻力，可通过 Tauri Sidecar 运行 Node.js 脚本作为降级方案
