# 01 - 项目概览与现状分析

## 1. 项目简介

AionUi 是一个跨平台 AI 代理协作工作平台，当前基于 Electron 构建。本次重构目标是将其迁移至 **Tauri 2.x**，后端完全使用 Rust 重写，前端在保持完全一致的界面和功能的前提下进行架构优化。

## 2. 现有技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面框架 | Electron 37 | 三进程模型（主进程/渲染/Worker） |
| 前端框架 | React 19 + TypeScript 5.8 | 严格模式 |
| UI 组件库 | Arco Design 2 | 企业级组件库 |
| 样式方案 | UnoCSS 66 | 原子化 CSS + 语义化 Token |
| 构建工具 | electron-vite 5 / Vite 6 | 多目标构建 |
| 后端运行时 | Node.js 22-24 | 主进程逻辑 |
| 数据库 | SQLite (better-sqlite3) | 本地持久化 |
| WebUI 服务器 | Express 5 + WebSocket | 远程访问 |
| 国际化 | i18next | 6 种语言 |
| 测试 | Vitest + Playwright | 单元 + E2E |
| 包管理 | Bun | 快速安装 |

## 5. 现有代码问题分析

### 5.1 前端重复模式
1. **Agent 聊天组件重复**：ACP、Codex、Gemini、Nanobot、OpenClaw 各有独立的 Chat/SendBox/Message 组件，大量逻辑重复
2. **设置页面模式重复**：每个设置页面结构相似，缺乏统一框架
3. **消息渲染重复**：不同 Agent 的消息组件存在大量相似代码
4. **Context + Hook 样板代码**：每个状态域都需要 Context → Provider → useXxx 三件套

### 5.2 架构层面
- Electron 主进程承载过多逻辑（36KB 入口文件）
- IPC Bridge 模块过多（29 个），部分职责边界模糊
- Worker 进程管理较为分散
- 文件系统桥接模块过大（50KB）

## 6. 迁移核心挑战

| 挑战 | 说明 | 应对策略 |
|------|------|---------|
| Node.js → Rust | 后端逻辑全部用 Rust 重写 | 模块化设计，逐步迁移 |
| Electron IPC → Tauri Commands | 29 个 IPC 桥接需要重新实现 | 统一 Command 接口设计 |
| npm 原生模块 | better-sqlite3、bcrypt 等 | Rust 原生替代（sqlx、argon2） |
| Worker 进程 | AI Agent Worker 需要新方案 | Tauri Sidecar / tokio 异步任务 |
| Express WebUI | 需要 Rust HTTP 服务器替代 | Axum / Actix-web |
| 子进程管理 | ACP/Codex 等外部进程 | Tauri Shell Plugin / tokio::process |
| 文件系统操作 | 大量 Node.js fs 操作 | Rust std::fs + tokio::fs |
