## 3. 模块间接口契约

### 3.1 Agent ↔ Chat

```rust
// Agent 完成消息后，通过 ChatService 持久化
impl AgentService {
    async fn on_agent_message(&self, conv_id: &str, event: AgentEvent) {
        match event {
            AgentEvent::MessageComplete { message } => {
                self.chat_service.add_message(conv_id, message).await;
            }
            // ...
        }
    }
}
```

### 3.2 Cron ↔ Agent

```rust
// Cron 任务触发时，创建临时对话并调用 Agent
impl CronService {
    async fn execute_job(&self, job: &CronJob) {
        let conv = self.chat_service.create(CreateChat {
            name: format!("Cron: {}", job.name),
            agent_type: job.agent_type.clone(),
            ..Default::default()
        }).await?;

        self.agent_service.send_message(
            &conv.id,
            &job.prompt,
            None,
            self.event_tx.clone(),
        ).await?;
    }
}
```

### 3.3 Channel ↔ Agent

```rust
// 通道消息到达时，路由到对应的 Agent
impl ChannelService {
    async fn on_channel_message(&self, session: &ChannelSession, content: &str) {
        let conv_id = session.chat_id.as_ref()
            .ok_or(AppError::NotFound("No linked chat"))?;

        self.agent_service.send_message(
            conv_id,
            content,
            None,
            self.event_tx.clone(),
        ).await?;
    }
}
```

### 3.4 WebUI ↔ Core Services

```rust
// WebUI 路由直接调用核心服务
async fn api_send_message(
    State(state): State<Arc<AppState>>,
    Json(input): Json<SendMessageInput>,
) -> Result<Json<()>, AppError> {
    state.agent_service.send_message(
        &input.chat_id,
        &input.content,
        input.files,
        state.event_bus.subscribe(),
    ).await?;
    Ok(Json(()))
}
```

## 4. 前端架构重构（核心改进）

### 4.0 原项目问题诊断

对原项目 AionUi 的深度分析发现两个关键架构问题：

#### 问题 1：会话组件大量重复

原项目为每种 Agent 类型维护独立的 Chat + SendBox 组件：

```
chat/
├── acp/AcpChat.tsx + AcpSendBox.tsx
├── codex/CodexChat.tsx + CodexSendBox.tsx
├── gemini/GeminiChat.tsx + GeminiSendBox.tsx
├── nanobot/NanobotChat.tsx + NanobotSendBox.tsx
└── openclaw/OpenClawChat.tsx + OpenClawSendBox.tsx
```

**5 个 Chat 组件结构几乎完全相同**（都是 `ChatProvider → MessageList → SendBox`），差异仅在于 `type` 字段和 SendBox 组件。

**5 个 SendBox 组件有 ~70% 重复逻辑**：
- 共有：草稿管理、文件附件、思想流显示、消息转换、事件系统、斜杠命令
- 差异：状态字段名、权限请求类型、Token 展示、Agent 特有功能

#### 问题 1b：SendBox 组件 ~50-60% 代码重复

5 个 SendBox 文件总计 **3159 行**：

| 文件 | 行数 |
|------|------|
| AcpSendBox.tsx | 679 |
| GeminiSendBox.tsx | 927 |
| NanobotSendBox.tsx | 406 |
| OpenClawSendBox.tsx | 630 |
| CodexSendBox.tsx | 517 |

**完全重复的逻辑**（~150 行/文件，占 25-30%）：
- Draft 状态管理（`mutateDraft`, `setContent`）
- 文件处理（`handleFilesAdded`, `handleWorkspaceFileDrop`）
- 事件监听注册（`sendbox.fill`, `selected.file`）
- JSX prefix 结构（`HorizontalFileList` + `FilePreview`）
- Slash 命令集成（`useSlashCommands`）

**模式相同但参数不同**（~200 行/文件，占 20-25%）：
- Thought 节流逻辑（50ms 防抖，4 文件完全相同）
- `responseStream` 事件处理（结构相同但 case 不同）
- 初始消息处理（`initialMessage` + `processedKey` 防重）

**各文件独有部分**（~150 行/文件，占 15-20%）：
- Gemini：模型选择、Agent 就绪检测、token 追踪、`AgentSetupCard`
- ACP：`AcpConfigSelector`、context usage
- OpenClaw：`validateRuntimeMismatch`、starOffice 安装检测
- Codex：`codex_model_info`、`hasContentInTurnRef`

#### 问题 1c：Modal/Drawer 状态管理泛滥

项目中 **131 处** 独立实现 Modal 可见性管理：
- 已有抽象工具 `ModalHOC.tsx`（89 行）但仅 3 个文件使用
- 已有 `AionModal.tsx`（380 行）但使用不统一
- 其余 Modal 各自手写 `useState(visible)` + `onCancel` + `onSubmit`

#### 问题 1d：权限确认逻辑分散

权限处理分散在 3 个独立位置：
- `MessageAcpPermission.tsx`（138 行）— ACP 权限 UI + 确认逻辑
- `ChatChatConfirm.tsx`（276 行）— 通用权限确认
- `MessageList.tsx` — `codex_permission` 返回 null（静默忽略）

三处有相似的状态管理：`hasResponded`、`isResponding`、`selected`。

#### 问题 1e：28 个 Bridge 文件的模板代码

IPC Bridge 层总计 **6000+ 行**，每个 Bridge 遵循相同的 provider 注册模式：
- `chatBridge.ts`（519 行）+ `acpChatBridge.ts`（319 行）+ `geminiChatBridge.ts`（27 行）功能重叠
- 所有 Bridge 都重复实现 error handling 和 response wrapper

> 在 Tauri 重构中，28 个 Bridge 将被 tauri-specta 自动生成的 bindings 完全替代，此问题自然消失。

#### 问题 1f：Context Provider 模式不一致

存在 3 种不同的 Context 创建方式：
- 手动 `createContext` + `Provider` + `useXxx` hook（AuthContext 210 行、ChatTabsContext 264 行）
- 通用工具函数 `createContext.tsx`（48 行）— 存在但仅少数文件使用
- 极简版（LayoutContext 18 行、settingsViewContext 11 行）

> 在 Tauri 重构中，大部分 Context 将被 Zustand Store 替代（见 02 文档 §1.2），剩余的会话级 Context 统一使用工厂函数。

#### 问题 2：工具渲染器缺乏通用映射机制

原项目为不同 Agent 类型维护**三套独立的工具渲染体系**：

| Agent 体系 | 渲染入口 | 工具类型定义 |
|-----------|---------|-------------|
| ACP | `MessageAcpToolCall.tsx` | `IMessageAcpToolCall` |
| Codex | `MessageCodexToolCall.tsx` → switch/case | `CodexToolCallUpdate` (discriminated union) |
| Gemini/通用 | `MessageToolGroup.tsx` (600+ 行) | `IMessageToolGroup` |

**问题**：
- 相同的工具（如 `exec_command`、`file_edit`）在不同 Agent 体系中有不同的渲染路径
- 新增工具类型需要修改 5 处代码（类型定义 → 事件数据 → union 扩展 → switch 路由 → 组件创建）
- 没有统一的注册表，无法动态扩展
- ACP 和 Codex 的工具渲染器无法复用

### 4.2 改进二：通用工具渲染器注册表（Registry 模式）

> **行业验证**：此模式与 **assistant-ui**（⭐8813，React AI Chat 领域事实标准）的 Tool UI Registry 高度一致。
> assistant-ui 的 `useToolUIs().getToolUI(toolName)` 和 `makeAssistantToolUI({ toolName, render })`
> 本质上就是 `Map<toolName, Component>` 注册表。其配套库 **tool-ui**（⭐557）进一步验证了
> 「为每种工具提供专用渲染组件 + 统一注册」是当前最流行的模式。
>
> **我们的方案额外优势**：前缀匹配（`exec_command_begin` → `exec_command`）和通配符（`mcp_*`），
> 这些是 assistant-ui 不具备的。assistant-ui 仅支持精确匹配，对于我们需要处理 `_begin/_end/_delta`
> 多阶段工具事件的场景不够灵活。

**核心思路**：用 `Map<string, ComponentType>` 注册表替代 switch/case 工厂，统一所有 Agent 的工具渲染。

```
原架构（3 套独立体系）          →        新架构（统一注册表）
┌─────────────────┐                   ┌───────────────────────┐
│ AcpToolCall     │                   │  ToolRendererRegistry │
│ (独立渲染逻辑)   │                   │                       │
├─────────────────┤                   │  Map<string, {        │
│ CodexToolCall   │  ──────────→      │    component,         │
│ (switch/case)   │                   │    icon,              │
├─────────────────┤                   │    label              │
│ ToolGroup       │                   │  }>                   │
│ (600行巨石组件)  │                   │                       │
└─────────────────┘                   │  .register()          │
                                      │  .resolve()           │
                                      │  .renderTool()        │
                                      └───────────────────────┘
```

#### 注册表核心实现

```typescript
// services/tool-renderer-registry.ts

interface ToolRendererEntry {
  /** 渲染组件 */
  component: React.ComponentType<ToolRenderProps>;
  /** 图标（emoji 或图标组件） */
  icon: string | React.ComponentType;
  /** 显示名称（支持 i18n key） */
  label: string;
  /** 支持的 Agent 类型（空 = 全部） */
  agents?: AgentType[];
  /** 匹配优先级（高优先级覆盖低优先级） */
  priority?: number;
}

/** 统一的工具渲染 Props（参考 assistant-ui ToolCallContentPartProps） */
interface ToolRenderProps<TArgs = Record<string, unknown>, TResult = unknown> {
  toolCallId: string;
  toolName: string;
  status: ToolStatus;
  args: TArgs;
  result?: TResult;
  /** Human-in-the-loop：允许用户在 UI 中提供工具结果（如审批、表单填写） */
  addResult?: (result: TResult) => void;
  /** 原始的 Agent 特有数据（向后兼容） */
  raw?: unknown;
}

type ToolStatus =
  | { type: 'running' }                              // 执行中
  | { type: 'complete' }                              // 已完成
  | { type: 'error'; error: string }                  // 出错
  | { type: 'requires-action'; reason: string };      // 需要用户操作

// ─────────────────────────────────────────────
// 底层存储：Class-based（高效查找 + 通配符/前缀匹配）
// ─────────────────────────────────────────────

class ToolRendererStore {
  private renderers = new Map<string, ToolRendererEntry>();
  private listeners = new Set<() => void>();
  readonly fallback: ToolRendererEntry = {
    component: GenericToolDisplay,
    icon: '⚙️',
    label: 'tool.generic',
  };

  register(toolName: string, entry: ToolRendererEntry): () => void {
    this.renderers.set(toolName, entry);
    this.notify();
    // 返回 unsubscribe 函数（React 生命周期友好）
    return () => {
      this.renderers.delete(toolName);
      this.notify();
    };
  }

  registerAll(entries: Record<string, ToolRendererEntry>): void {
    for (const [name, entry] of Object.entries(entries)) {
      this.renderers.set(name, entry);
    }
    this.notify();
  }

  resolve(toolName: string, agentType?: AgentType): ToolRendererEntry {
    // 1. 精确匹配
    const exact = this.renderers.get(toolName);
    if (exact && this.matchesAgent(exact, agentType)) return exact;

    // 2. 前缀匹配（如 "exec_command_begin" → "exec_command"）
    const prefix = this.findPrefixMatch(toolName);
    if (prefix && this.matchesAgent(prefix, agentType)) return prefix;

    // 3. 通配符匹配（如 "mcp_*"）
    const wildcard = this.findWildcardMatch(toolName);
    if (wildcard && this.matchesAgent(wildcard, agentType)) return wildcard;

    return this.fallback;
  }

  subscribe(listener: () => void): () => void {
    this.listeners.add(listener);
    return () => this.listeners.delete(listener);
  }

  private notify() {
    this.listeners.forEach((l) => l());
  }

  private matchesAgent(entry: ToolRendererEntry, agentType?: AgentType): boolean {
    if (!entry.agents || entry.agents.length === 0) return true;
    return agentType ? entry.agents.includes(agentType) : true;
  }

  private findPrefixMatch(toolName: string): ToolRendererEntry | undefined {
    const parts = toolName.split('_');
    while (parts.length > 1) {
      parts.pop();
      const match = this.renderers.get(parts.join('_'));
      if (match) return match;
    }
    return undefined;
  }

  private findWildcardMatch(toolName: string): ToolRendererEntry | undefined {
    for (const [pattern, entry] of this.renderers) {
      if (pattern.endsWith('*') && toolName.startsWith(pattern.slice(0, -1))) {
        return entry;
      }
    }
    return undefined;
  }
}

export const toolRendererStore = new ToolRendererStore();

// ─────────────────────────────────────────────
// React 层：Hook-based（React-idiomatic 访问方式）
// ─────────────────────────────────────────────

/** 查询工具渲染器（响应式，注册表变化时自动更新） */
function useToolRenderer(toolName: string, agentType?: AgentType): ToolRendererEntry {
  return useSyncExternalStore(
    toolRendererStore.subscribe.bind(toolRendererStore),
    () => toolRendererStore.resolve(toolName, agentType),
  );
}

/** 动态注册工具渲染器（组件生命周期内有效，卸载自动注销） */
function useRegisterToolRenderer(toolName: string, entry: ToolRendererEntry): void {
  useEffect(() => {
    return toolRendererStore.register(toolName, entry);
  }, [toolName]);
}
```

#### 改进六：Portal 注入 Hook（usePortalInjection）

**问题**：6 个文件重复 70-80% 的 Portal 注入代码（~600 行）。

**方案**：提取 `usePortalInjection` Hook。

```typescript
// hooks/usePortalInjection.ts

interface UsePortalInjectionOptions {
  /** 容器 ref */
  containerRef: React.RefObject<HTMLElement>;
  /** 注入目标选择器 */
  targetSelector: string;
  /** 要注入的组件 */
  children: React.ReactNode;
  /** 依赖项（变化时重新注入） */
  deps?: React.DependencyList;
}

function usePortalInjection({
  containerRef,
  targetSelector,
  children,
  deps = [],
}: UsePortalInjectionOptions): void {
  const portalRef = useRef<HTMLDivElement | null>(null);

  if (!portalRef.current) {
    portalRef.current = document.createElement('div');
  }

  useLayoutEffect(() => {
    const container = containerRef.current;
    if (!container) return;

    const target = container.querySelector(targetSelector);
    if (target && portalRef.current) {
      if (!target.contains(portalRef.current)) {
        target.appendChild(portalRef.current);
      }
    }

    return () => {
      if (portalRef.current && portalRef.current.parentNode) {
        portalRef.current.parentNode.removeChild(portalRef.current);
      }
    };
  }, [containerRef, targetSelector, ...deps]);

  // 渲染 Portal
  useEffect(() => {
    if (portalRef.current) {
      const root = ReactDOM.createRoot(portalRef.current);
      root.render(children);
      return () => root.unmount();
    }
  }, [children]);
}
```
