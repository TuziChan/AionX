# 08 - 目录结构设计

## 1. 原项目结构分析

### 1.1 原项目统计

| 指标 | 数值 | 说明 |
|------|------|------|
| 总代码行数 | 23,636 LOC | 中型代码库 |
| TypeScript/TSX 文件 | 626 个 | — |
| 最大组件 | renderer (58,239 LOC) | 占 58% |
| 最大子组件 | conversation (18,541 LOC) | 占 renderer 的 32% |
| Agent 数量 | 5 个 | 高重复风险 |
| 平均文件大小 | 38 LOC | 模块化良好 |

### 1.2 原项目架构问题

#### 问题 1：按类型组织 vs 按功能组织混乱

```
src/
├── renderer/
│   ├── pages/           # 按功能（页面）
│   │   ├── conversation/
│   │   │   ├── acp/     # 按 Agent 类型
│   │   │   ├── gemini/  # 按 Agent 类型
│   │   │   └── ...
│   ├── components/      # 按类型（组件）
│   ├── hooks/           # 按类型（Hook）
│   └── utils/           # 按类型（工具）
├── agent/               # 按功能（Agent）
│   ├── acp/
│   ├── gemini/
│   └── ...
└── channels/            # 按功能（频道）
```

**问题**：
- `renderer/pages/conversation/` 按 Agent 类型分目录，导致 5 个重复的 Chat/SendBox
- `agent/` 目录按 Agent 类型分，但每个 Agent 内部又按类型分（connection/detector/store）
- 共享组件散落在 `components/`、`base/`、`messages/` 多个位置

#### 问题 2：深层嵌套

```
src/renderer/pages/conversation/preview/components/viewers/MarkdownViewer.tsx
                                                    ^^^^^^^ 6 层深度
```

#### 问题 3：职责不清

- `src/common/` — 既有类型定义，又有业务逻辑（chatLib.ts 288 行）
- `src/shared/` — 与 `common/` 职责重叠
- `src/types/` — 与 `common/` 的类型定义重叠

## 2. 新项目目录结构（Tauri 2.x）

### 2.1 设计原则

1. **功能优先，类型辅助** — 核心功能按领域组织，通用工具按类型组织
2. **扁平化** — 避免超过 4 层嵌套
3. **单一职责** — 每个目录有明确的职责边界
4. **可预测性** — 开发者能快速定位文件位置

## 3. 关键设计决策

### 3.1 功能模块（features/）设计

**原则**：每个功能模块是一个自包含的领域，包含该领域的所有内容。

```
features/chat/
├── components/      # 该功能的 UI 组件
├── adapters/        # 该功能的适配器（Agent 适配器）
├── hooks/           # 该功能的 Hooks
├── stores/          # 该功能的状态管理
├── types/           # 该功能的类型定义
├── routes.tsx       # 该功能的路由
└── index.ts         # 导出接口
```

**优势**：
- 高内聚：相关代码在一起
- 低耦合：功能间通过明确的接口交互
- 可移植：整个功能可以作为一个单元移动/删除
- 可测试：功能的所有依赖都在同一目录

### 3.2 共享组件（components/）设计

**原则**：只放真正跨功能复用的组件。

```
components/
├── base/            # 基础组件（Arco Design 封装）
├── ui/              # 通用 UI 组件（Markdown、Diff、Thought 等）
└── layout/          # 布局组件
```

**判断标准**：
- 被 3+ 个功能模块使用 → 放 `components/`
- 仅被 1-2 个功能使用 → 放功能模块内部
- 功能特定的组件 → 永远不放 `components/`

### 3.3 消息渲染系统独立

**原因**：消息渲染是一个复杂的子系统，包含：
- 11 种消息类型
- 工具渲染注册表
- 7+ 种工具渲染器

**设计**：作为独立的 `features/messages/` 模块，而非放在 `conversation/` 内部。

### 3.4 Rust 后端分层

```
src-tauri/src/
├── db/              # 数据访问层（Repository）
├── models/          # 数据模型（带 specta::Type）
├── services/        # 业务逻辑层
├── agents/          # Agent 实现（统一 trait）
├── commands/        # Tauri Commands（API 层）
└── events/          # Tauri Events 定义
```

**分层职责**：
- `db/` — 只负责数据库操作，不包含业务逻辑
- `models/` — 纯数据结构，带 `specta::Type` 用于类型导出
- `services/` — 业务逻辑，协调多个 Repository
- `commands/` — 薄层，仅做参数验证和调用 Service

## 4. 与原项目对比

| 维度 | 原项目 | 新项目 | 改进 |
|------|--------|--------|------|
| **组织方式** | 按类型 + 按功能混合 | 功能优先，类型辅助 | 更清晰 |
| **聊天组件** | 5 个独立目录（acp/gemini/...） | 1 个 chat/ + adapters/ | 消除重复 |
| **消息渲染** | 分散在 messages/ 和 conversation/ | 独立的 messages/ 模块 | 职责清晰 |
| **共享组件** | components/、base/、messages/ 混乱 | components/ 统一管理 | 可预测 |
| **最大嵌套深度** | 6 层 | 4 层 | 更扁平 |
| **类型定义** | common/、shared/、types/ 重叠 | types/ 统一 | 单一职责 |
| **Hooks** | 分散在各处 | 功能内 + 全局 hooks/ | 易查找 |
| **Stores** | Context 为主 | Zustand，功能内 + 全局 | 现代化 |
| **命名规范** | conversation（正式） | chat（行业标准） | 符合趋势 |

## 5. 迁移路径

### 5.1 第一阶段：基础设施

1. 创建 Tauri 项目骨架
2. 配置 tauri-specta
3. 实现 Rust 后端基础层（db/models/services）

## 6. 文件命名规范

### 6.1 React 组件

- **页面组件**：`PascalCase.tsx`（如 `ChatSession.tsx`）
- **UI 组件**：`PascalCase.tsx`（如 `MessageItem.tsx`）
- **布局组件**：`PascalCase.tsx`（如 `MainLayout.tsx`）

### 6.2 Hooks

- **格式**：`use + PascalCase.ts`（如 `useSendBoxCore.ts`）
- **位置**：功能内 `hooks/` 或全局 `hooks/`

### 6.3 Stores

- **格式**：`camelCase + Store.ts`（如 `chatStore.ts`）
- **位置**：功能内 `stores/` 或全局 `stores/`

### 6.4 Services

- **格式**：`camelCase.ts`（如 `chat.ts`）
- **位置**：前端 `services/`，后端 `services/`

### 6.5 Types

- **格式**：`camelCase.ts` 或 `index.ts`
- **位置**：功能内 `types/` 或全局 `types/`

### 6.6 Rust 文件

- **格式**：`snake_case.rs`（如 `chat_service.rs`）
- **模块**：每个目录有 `mod.rs`
