# 前端统一重构蓝图

本文档用于冻结 AionX 全前端一次性重构的基础决策。它不是执行清单，而是执行清单背后的“统一 UI 框架、统一布局体系、统一组件风格”规范。后续所有前端页面与共享组件重构，都必须优先遵循本文档；如果计划发生变化，需要同时更新根目录 `plans.md`。

## 1. 目标

本次重构不是“把设置页换一套皮肤”，而是把整个前端改造成一套统一的桌面工作台界面。新界面要满足四个目标：

1. 功能不退化。现有 Tauri 命令、typed API、路由和 smoke 基线继续保留。
2. 设计系统统一。登录页、主导航、聊天页、设置页、工具页使用同一套 token、同一套组件、同一套布局语言。
3. 桌面优先，但移动与窄屏可用。AionX 是桌面应用，因此视觉重心应偏桌面工作台，而不是移动优先的营销站。
4. 基础优先。先搭建 design system 和 page shell，再做页面内容迁移，不允许反过来一页一页临时拼样式。

## 2. 推荐技术栈

这是当前最适合 Tauri 桌面 React 应用的一条主流实现路线：

- 运行时：`Tauri 2 + React 19 + Vite 7 + React Router 7`
- UI 框架：`Tailwind CSS v4 + shadcn/ui + Radix UI`
- 类名与变体：`clsx + tailwind-merge + class-variance-authority`
- 图标：`lucide-react`
- 表单：`react-hook-form + zod + @hookform/resolvers`
- 异步状态：`@tanstack/react-query`
- 本地状态：`zustand`
- 动画：优先 CSS transitions，必要时引入 `motion`，但不作为首批基础依赖

这条路线适合当前项目的原因很明确：

- `shadcn/ui` 适合做“应用自己的组件库”，而不是被第三方主题绑死。
- `Tailwind CSS v4` 的 theme variables 很适合把现有 CSS variables 迁成统一 token。
- `Radix UI` 适合桌面应用常见的菜单、弹层、sheet、popover、tabs、tooltip 等无障碍 primitives。
- `React Hook Form + Zod` 非常适合设置页和复杂表单。
- `TanStack Query` 适合前端存在大量异步加载、保存和刷新动作的设置/工作区页面。

## 3. 统一架构

采用轻量领域分层，而不是继续把代码按 `components/hooks/utils` 全局分桶。

目标目录如下：

    src/
      app/
        layouts/
        providers/
        router/
      pages/
      widgets/
      features/
      entities/
      shared/
        ui/
        lib/
        hooks/
        styles/
        config/

职责边界如下：

- `app`：应用入口、全局 provider、全局 layout、路由组装。
- `pages`：路由级页面组合。
- `widgets`：大块 UI 结构件，例如 App Sidebar、Conversation Header、Settings Navigation。
- `features`：与业务动作强相关的可复用片段，例如 Assistant Editor、Tool Server Editor、Channel Config Form。
- `entities`：领域模型与小型领域视图，例如 `assistant`、`model-provider`、`tool-server`。
- `shared/ui`：shadcn primitives 与基础 variants。
- `shared/lib`：`cn()`、格式化、键盘工具、平台判断等。
- `shared/styles`：全局样式入口、token、theme、动画。

## 4. 样式分层

新前端只允许保留四层样式：

1. `tokens layer`
   负责颜色、字号、间距、圆角、阴影、边框、动画曲线。
2. `primitive layer`
   负责 shadcn 基础组件，如 button、input、tabs、dialog、sheet。
3. `semantic layer`
   负责业务语义组件，如 SectionCard、PageToolbar、FieldRow、StatusBadge。
4. `page layer`
   只负责页面组合，不再定义新的全局基础按钮或输入框规则。

不再允许：

- 页面直接覆盖第三方组件大量内部类名
- 在单个超大 CSS 文件中混合所有页面和组件规则
- 同一类按钮在不同页面各自定义高度和圆角

## 5. 设计 Token

### 5.1 颜色

整体方向是 `graphite + cobalt`，不再沿用旧项目偏紫色主调。

- 中性底色：graphite / zinc / slate 一组，负责背景、边框和层级
- 主强调色：cobalt blue
- 成功色：green
- 警告色：amber
- 危险色：red
- 信息色：blue

语义 token 至少包括：

- `--background`
- `--foreground`
- `--card`
- `--card-foreground`
- `--muted`
- `--muted-foreground`
- `--accent`
- `--accent-foreground`
- `--primary`
- `--primary-foreground`
- `--border`
- `--input`
- `--ring`
- `--destructive`
- `--destructive-foreground`

### 5.2 间距

只保留一组主间距尺度：

- `4`
- `6`
- `8`
- `12`
- `16`
- `24`
- `32`
- `40`

页面和组件都从这组尺度中取值，不再出现大量 `18px`、`22px`、`28px` 这类游离值，除非属于明确的视觉 token。

### 5.3 圆角

统一圆角尺度：

- `sm = 10px`
- `md = 14px`
- `lg = 18px`
- `xl = 24px`
- `pill = 999px`

### 5.4 阴影

只保留三档：

- `shadow-sm`：轻微抬升
- `shadow-md`：卡片/弹层
- `shadow-lg`：dialog/sheet/popover 的浮层强调

所有大面积夸张阴影一律禁用。

### 5.5 字体

推荐：

- 主字体：`Geist Sans`
- 等宽字体：`Geist Mono`
- 中文回退：`PingFang SC`, `Microsoft YaHei`, `Noto Sans SC`

字体层级：

- 页面标题：`30-36`
- 区块标题：`18-20`
- 正文：`14-15`
- 次要说明：`12-13`

## 6. 页面整体布局

### 6.1 AppFrame

这是所有登录后页面共享的总壳层。

桌面端结构：

- 自定义 titlebar
- 左侧主导航
- 主内容区
- 可选的右侧辅助面板

移动/窄屏结构：

- 顶栏
- 左侧导航改为 drawer
- 主内容区单列滚动

推荐尺寸：

- titlebar 高度：`40`
- sidebar 展开宽度：`280`
- sidebar 收起宽度：`72`
- page gutter：`24`

### 6.2 SettingsFrame

设置页不是普通内容页，必须有独立的二级导航结构。

桌面端：

- 左侧二级导航
- 右侧内容区
- 内容区顶部页标题和说明
- 页面内部只允许一种主滚动容器

移动端：

- 顶部返回与标题
- 横向 section switcher 或 drawer section nav
- 内容区单列滚动

宽度语义固定：

- `form = 720px`
- `content = 960px`
- `wide = 1200px`

### 6.3 ConversationFrame

聊天页是产品主场景，布局必须比设置页更强势。

桌面端：

- 左侧会话/工作区
- 中央主聊天区
- 右侧可折叠辅助区（预览、workspace、details）

移动端：

- 主聊天区优先
- 左右侧面板改为 sheet/drawer

### 6.4 FormPageFrame

用于登录页、关于页、简化表单页、说明页。

规则：

- 内容居中但不空
- 明确主卡片和辅文层级
- 表单页优先窄栏阅读体验

## 7. 组件风格

### 7.1 按钮

统一按钮变体：

- `primary`
- `secondary`
- `outline`
- `ghost`
- `destructive`
- `link`

统一按钮尺寸：

- `sm = 32`
- `md = 36`
- `lg = 40`
- `icon = 36`

要求：

- 不同页面不再自己定义按钮高度
- 主按钮用实色，次按钮用 outline 或 ghost
- 图标按钮必须和普通按钮同一圆角体系

### 7.2 卡片

统一卡片只分三类：

- `surface card`：普通信息卡
- `section card`：设置区块与工作区区块
- `elevated panel`：浮层/强调面板

规则：

- 卡片靠边框、背景层次和弱阴影区分
- 不用“大色块 hero 卡片”到处抢注意力

### 7.3 表单

统一表单结构：

- label
- description（可选）
- control
- message / error

设置页优先使用 `FieldRow` 或 `FieldStack` 两种模式，不允许每页再定义自己的表单骨架。

### 7.4 导航

统一导航分三层：

- 主导航：应用一级模块
- 次导航：Settings sections、Conversation tabs、工具分组
- 局部标签：Tabs、segmented controls、filters

所有导航 active 态必须遵循同一视觉规则：背景轻强调 + 前景加深 + icon 同步变化。

### 7.5 弹层

统一弹层策略：

- 小交互：Popover / DropdownMenu
- 中等交互：Sheet
- 高风险或确认：Dialog

不再混用多个完全不同风格的 Drawer/Dialog。

### 7.6 空状态与反馈

统一反馈组件：

- `EmptyState`
- `LoadingState`
- `ErrorState`
- `SuccessBanner`
- `StatusBadge`

不能继续每页自己写一套空状态布局。

## 8. 动效

动效只保留有意义的三类：

- 页面进入淡入上浮
- Drawer / Sheet 滑入
- Hover elevation / active press

禁止：

- 大面积持续动画背景
- 与桌面工作台无关的夸张 bounce 动效
- 页面之间完全不同的动画语法

## 9. 基础落地顺序

第一优先级必须是基础，而不是页面。

顺序固定如下：

1. 安装 Tailwind v4、shadcn、Radix 与基础依赖
2. 建立 `components.json`
3. 建立 `shared/styles` 的 tokens / theme / globals
4. 建立 `shared/ui` primitives
5. 建立 `AppFrame / SettingsFrame / ConversationFrame / FormPageFrame`
6. 建立语义化共享组件
7. 才开始页面重写

## 10. 不该做的事

- 不要一开始就逐页重写视觉，基础还没定时这样只会返工
- 不要把 shadcn 当成“装一个库就结束”，它本质上是你自己的组件代码
- 不要在新体系里继续复制旧的 Arco class 覆盖方式
- 不要让 settings 成为唯一例外，整站必须同一套设计系统
- 不要为了快，重新造一个超大的 `app-shell.css`

## 11. 当前结论

后续重构应按以下原则执行：

- 功能层尽量保留，展示层系统性重写
- 新 UI 必须先做基础设施和 design system
- 所有页面最终落入同一套 frame + token + component 体系
- 最终交付不保留长期新旧混合 UI
