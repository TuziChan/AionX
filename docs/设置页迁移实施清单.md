## 1. 目标与结论
本文件定义 `AionX` 设置页的最终目标架构、页面规范与实施顺序。

唯一结论如下：

1. 设置页是 `AionX` 的一级功能模块，不再视作 `AionUi` modal 内容的 page 化包装。
2. `layout-route` 是唯一允许的承载方案：`AppLayout` 与 `SettingsLayout` 平级，`/settings -> SettingsLayout -> Outlet -> page`。
3. 不再为向后兼容保留 page wrapper、modal/page 双态判断、AionUi 文案、主工作区侧栏复用或旧式泛化设置读写方式。
4. 所有页面、样式、导航、状态和后端接口都围绕 `AionX` 自身产品语义重建。

本方案追求的是规范、清晰、可维护，而不是“尽量少改”。

---

## 2. 非目标
以下内容不属于本次迁移目标，且应在实施中主动避免：

- 不再保留“每个页面内部包一层 `SettingsPageWrapper`”的结构。
- 不再保留 `SettingsViewModeProvider` / `modal | page` 双模式分支。
- 不再把 settings 专用导航挂在全局 `Sidebar` 的二级状态里。
- 不再在 `/settings/*` 下复用主工作区的聊天历史、快捷入口和底部操作区。
- 不再以 `AionUi` 的目录结构、组件命名、文案内容作为兼容前提。
- 不再把 `getSettings/updateSettings` 作为设置页核心业务接口的长期方案。
- 不再使用 `window.prompt`、原始 JSON 文本框、占位按钮等临时交互作为最终实现。

## 3. 最终架构决策

### 3.1 路由结构
设置页采用与主工作区平级的独立 shell：

```tsx
RootLayout
|- AppLayout
|  |- /guid
|  |- /conversation/:id
|  |- /cron
|  |- ...
|- SettingsLayout
   |- /settings/gemini
   |- /settings/model
   |- /settings/agent
   |- /settings/display
   |- /settings/webui
   |- /settings/system
   |- /settings/tools
   |- /settings/about
   |- /settings/ext/:tabId
```

约束如下：

- `SettingsLayout` 是独立于 `AppLayout` 的设置工作区壳层。
- `SettingsLayout` 拥有 settings 模块的导航、页面标题、移动端切换、错误态和空态承载职责。
- 各子页面只负责本页面的内容与交互，不再负责导航壳层。
- 扩展 tab 仍使用 `ext/:tabId` 路由，但 tab 注入由 settings 模块内部注册表统一处理。
- `/settings/*` 进入后不再显示主工作区完整侧栏。

### 3.2 导航归属
导航归属做如下重构：

- `AppLayout` 只负责主工作区导航。
- `SettingsLayout` 只负责设置工作区导航。
- 二者不能同时显示两套完整侧栏。

桌面端规则：

- 进入 `/settings/*` 后切换为独立 settings shell。
- 左侧只显示 settings 导航列。
- 左上显示“返回应用”入口。
- 不显示聊天历史、批量操作、新建会话等主工作区元素。

移动端规则：

- 进入 `/settings/*` 后不显示桌面式左侧导航。
- 顶部显示返回应用按钮、当前页面标题、可选搜索/操作按钮。
- 设置分类通过顶部横向 tabs 或分组列表切换。
- 页面内容为单列滚动布局。

返回位置规范：

- 桌面端返回入口位于 settings shell 顶部左上区域，可放在 `SettingsSidebar` 顶部或 `SettingsTopbar` 左侧。
- 移动端返回入口位于 `SettingsTopbar` 左上。
- 底部禁止放“返回应用”或“返回上一层”。
- 页面底部仅允许放保存、取消、重置、测试连接等页面操作。

这是 layout-route 架构下更规范的职责划分：

- 主工作区导航归 `AppLayout`。
- 设置导航归 `SettingsLayout`。
- 页面内容归子路由。

### 3.3 页面模式
设置页被定义为纯页面模块，因此：

- 移除 `settingsViewContext.tsx` 的业务意义。
- 不再为 modal 兼容保留 `page` / `modal` 分支。
- `SettingsPageWrapper.tsx` 不再作为独立包装器存在；其职责并入 `SettingsLayout` 与共享页面容器。
- 进入设置时整体切换工作区，而不是在主工作区内容区中嵌套一个设置页。

### 3.4 Shell 形态
最终 shell 形态参考 Codex 桌面版：

- 桌面端：`SettingsSidebar + SettingsContent`
- 移动端：`SettingsTopbar + SettingsTabs + SettingsContent`

禁止形态：

- `MainSidebar + SettingsSidebar + Content`
- `MainSidebar + MainContent(其中再嵌 SettingsNav + Page)`
- 页面内部自行模拟左侧导航

## 5. 前端分层规范

### 5.1 Layout 层
`SettingsLayout` 必须承担以下职责：

- 读取 settings 注册表并生成导航。
- 负责桌面端 settings 左侧导航。
- 负责移动端顶部返回栏与顶部 tabs。
- 根据当前路由显示页面标题、副标题、状态提示。
- 承载 `<Outlet />`。
- 统一处理 loading、error、not-found 和 extension tab 缺失等上层状态。
- 统一处理“返回应用”的导航动作。

`SettingsLayout` 不负责：

- 页面内表单字段逻辑。
- 页面内业务数据写回。
- 页面内弹窗状态。

### 5.2 Registry 层
新增 `settingsRegistry.ts`，作为 settings 模块唯一导航与页面元信息来源。

每个 tab 至少包含：

- `id`
- `path`
- `label`
- `icon`
- `title`
- `description`
- `widthPreset`
- `layoutMode`
- `order`
- `mobileVisible`

推荐结构：

```ts
type SettingsWidthPreset = 'narrow' | 'regular' | 'wide' | 'full';
type SettingsLayoutMode = 'form-stack' | 'split-view' | 'tabbed' | 'host';
```

扩展 tab 注入规则：

- 内置 tab 先注册。
- 扩展 tab 通过 `extensionRegistry.ts` 读取后合并。
- 合并逻辑由 registry 负责，不允许散落在 `Sidebar`、页面组件或样式文件中。

### 5.3 数据加载层
前端数据流采用“路由加载 + 页面局部交互状态”的方式：

- 路由进入时通过 page loader 或页面初始化 hook 拉取快照。
- 页面持久化数据以服务端快照为准。
- 组件 state 只保存草稿、展开态、选中态、弹窗态等临时状态。
- 修改完成后通过领域 API 提交，再局部更新或重新拉取。

禁止事项：

- 页面组件直接 `commands.xxx()`。
- 一个页面既负责请求、又负责数据变换、又负责大段展示。
- 把 localStorage 当作设置真源。

### 5.4 共享组件层
共享组件只提供结构与视觉，不承担领域逻辑。

必须保留或新增的共享组件：

- `SettingsPageFrame`
- `SettingsSectionCard`
- `SettingsSplitView`
- `SettingsEmptyState`
- `SettingsLoadingState`
- `SettingsErrorState`
- `PreferenceRow`

应移除或重命名的旧组件：

- `SettingsPage.tsx` 作为旧式 section/card 渲染器，不再作为最终页面基类。
- `settingsConfig.ts` 迁移为 registry 后不再保留旧定义。

## 6. 后端接口规范

### 6.1 总体原则
后端接口按领域拆分，不再以 `get_settings(category)` / `update_settings(category, settings)` 作为主业务协议。

最终形态要求：

- 每个页面至少有一组专属命令。
- 命令返回 typed 结构，而不是页面自行拼装 `JsonValue`。
- 后端完成校验、默认值填充和持久化。
- 前端不承担复杂 schema 修复逻辑。

### 6.2 泛化设置接口的定位
现有 `get_settings` / `update_settings` 只允许保留为：

- 迁移期兜底。
- 低价值、全局性、小体量配置项。

以下页面禁止继续把泛化设置接口作为最终主通道：

- Gemini
- Model
- Agent
- Tools
- WebUI
- Extension

### 6.3 建议的领域命令
建议新增或收口为以下命令族：

- Gemini: `get_gemini_settings`, `save_gemini_settings`, `get_google_auth_status`, `start_google_auth`, `logout_google_auth`

- Agent: `list_builtin_assistants`, `list_custom_assistants`, `create_custom_assistant`, `update_custom_assistant`, `delete_custom_assistant`, `save_builtin_assistant_preferences`
- Display: `get_display_settings`, `save_display_settings`
- WebUI: `get_webui_settings`, `get_webui_status`, `start_webui`, `stop_webui`, `change_webui_password`, `reset_webui_password`

### 7.1 统一样式骨架
settings 模块必须采用统一壳层类名，不再依赖零散的旧类：

- `.settings-layout`
- `.settings-layout__sidebar`
- `.settings-layout__main`

- `.settings-layout__header`
- `.settings-layout__body`
- `.settings-layout__content`
- `.settings-mobile-tabs`
- `.settings-mobile-tabs__item`
- `.settings-back-link`
- `.settings-page`

- `.settings-page--wide`
- `.settings-page--full`
- `.settings-section-card`
- `.settings-split-view`
- `.settings-split-view__list`
- `.settings-split-view__detail`

旧类如 `.settings-page-shell`、`.settings-card`、`.settings-card__body` 可在迁移期复用，但最终应收敛到新的模块命名。

### 7.2 宽度规范
页面宽度统一由 registry 决定，不允许每页自行写死：

- `narrow`: 在 `regular` 内容容器内部再缩窄的阅读型内容区，适用于 `About` 这类信息页。当前约束是内部窄栏保持约 `500px`，但页面仍挂在统一 settings shell 下。
- `regular`: 默认单列设置页宽度，适用于 `Gemini`、`Display`、`System` 以及多数表单型页面。桌面端以统一内容容器为基线，不要求每页自行再定义新的外层 max-width。
- `wide`: 复杂管理页的内容宽度语义，适用于 `Model`、`Agent`、`Tools`、`WebUI`。它表达的是“允许更多横向信息密度”，不代表每页都要用完全不同的外层 wrapper。
- `full`: 宿主页或沉浸式内容区，适用于 `Extension host` 等需要尽量释放横向空间的页面。

### 7.3 页面模板
settings 页面只允许使用以下四种模板：

- `form-stack`
  适用于表单与配置卡片纵向堆叠页。
- `split-view`
  适用于列表 + 详情的主从页。
- `tabbed`
  适用于单路由内部仍有二级分组的页。
- `host`
  适用于嵌入外部 UI 宿主页。

### 7.4 移动端规范
移动端必须满足：

- 进入设置后切换为独立移动端 settings shell。
- 顶部必须保留返回应用入口。
- 保留模块顶部横向 tabs 或等价分类切换器。
- 页面内容与桌面侧栏解耦，不依赖全局 `Sidebar` 的展开状态。
- 列表/详情类页面在移动端退化为单列堆叠、抽屉或独立详情页，不允许挤压成桌面双栏。
- 所有按钮尺寸、点击区域和间距满足手指操作场景。
- 顶部返回、标题、tabs、页面内容之间保持稳定层级，不允许多个 sticky 区域互相挤压。

### 7.5 组件风格基线
自 2026-03-18 17:35 +08:00 起，settings 页的共享组件风格不再追求与 `AionUi` 完全一致，而是按 AionX 自己的统一体验收敛。当前基线如下：

- 技术边界：继续使用 `React + Arco Design + UnoCSS`，通过 settings 作用域样式和 shared primitive 统一观感；当前阶段不引入 `shadcn/ui`，避免在同一模块内并存两套设计系统。
- 卡片：统一使用受控圆角、边框和阴影强度；同级 section card 不再各自定义完全不同的 padding。
- 表单：输入框、选择器、开关、slider、说明文本与危险提示统一节奏，优先通过 `SettingsField`、`PreferenceRow` 或后续 shared primitive 收敛。
- 按钮：主按钮、次按钮、幽灵按钮、图标按钮必须有固定尺寸档位，避免继续直接使用 Arco 默认观感造成每页按钮气质不一致。
- 信息层级：页面标题、区块标题、辅助说明、字段标签、危险动作和空状态文案必须形成稳定层级，不再让每页自己定义一套标题密度。
- 页面类型：至少形成三类稳定模板并优先覆盖代表页。
  - 信息页：`About`
  - 复杂表单页：`WebUI`
  - 管理页：`Tools`

## 13. 最终判断标准
如果某个实现满足以下任一情况，则视为不符合本规范：

- 页面内部仍自行包 wrapper。
- settings 导航仍挂在全局 `Sidebar` 的 settings 模式分支。
- `/settings/*` 仍显示主工作区聊天历史或主工作区侧栏完整内容。

- 页面组件直接写 `commands.xxx()`。
- 页面核心能力仍依赖 mock、prompt、空按钮或 JSON 主交互。
- 文案仍以 AionUi 为参照对象。

本次迁移完成的标志，不是“页面看起来差不多”，而是：

> `AionX` 设置页成为一个由 `layout-route` 驱动、边界清晰、页面规范统一、后端接口按领域收口的独立模块。
